# Hyperledger Fabric Network

BSC runs on a permissioned Hyperledger Fabric 2.5 network with 3 peer organizations and a single ordering service.

---

## Network Topology

| Component | MSP ID | Host | Port |
|---|---|---|---|
| Orderer | OrdererMSP | `orderer.bsc.gov` | 7050 |
| IT Dept peer | ITDeptMSP | `peer0.itdept.bsc.gov` | 7051 |
| Registrar peer | RegistrarMSP | `peer0.registrar.bsc.gov` | 8051 |
| MCA peer | MCAMSP | `peer0.mca.bsc.gov` | 9051 |

All peers and orderer have TLS enabled. Crypto material is generated by `cryptogen` and committed to the repo for development convenience — regenerate for production.

---

## Channel

- **Channel:** `bsc-channel`
- **Endorsement policy:** MAJORITY (2 of 3 orgs must endorse every write)
- **State database:** LevelDB (not CouchDB — lower RAM on dev machines)

The MAJORITY endorsement policy means no single ministry can write to the ledger unilaterally. Any write requires both the IT Dept node and at least one of Registrar or MCA to sign.

---

## Orderer

The current orderer is a **solo orderer** — suitable for development and demonstration. For production deployment, replace with a Raft-based ordering service (3 or 5 orderer nodes for Byzantine fault tolerance).

The solo orderer is configured at `blockchain/network/docker-compose-fabric.yml`. Raft migration is tracked in the roadmap.

## Deploying Chaincode

The deployment script is idempotent — safe to re-run at the same version:

```bash
# Deploy / redeploy all 4 chaincodes
bash blockchain/scripts/deploy-chaincode.sh

# Deploy a single chaincode at a new version
CC_VERSION=1.2 CC_SEQUENCE=3 CHAINCODES="property" bash blockchain/scripts/deploy-chaincode.sh
```

When upgrading: increment both `CC_VERSION` and `CC_SEQUENCE` together.

Current sequence numbers:
- `anomaly`: sequence 2
- `property`: sequence 2 (v1.2 ready at sequence 3)
- `access`: sequence 1
- `zkp`: sequence 2

## Manual Chaincode Queries

```bash
docker exec \
  -e CORE_PEER_ADDRESS=peer0.itdept.bsc.gov:7051 \
  -e CORE_PEER_LOCALMSPID=ITDeptMSP \
  -e CORE_PEER_MSPCONFIGPATH=/opt/gopath/src/github.com/hyperledger/fabric/peer/crypto/peerOrganizations/itdept.bsc.gov/users/Admin@itdept.bsc.gov/msp \
  -e CORE_PEER_TLS_ROOTCERT_FILE=/opt/gopath/src/github.com/hyperledger/fabric/peer/crypto/peerOrganizations/itdept.bsc.gov/peers/peer0.itdept.bsc.gov/tls/ca.crt \
  bsc-cli peer chaincode query --channelID bsc-channel --name anomaly \
  -c '{"function":"GetCitizenNode","Args":["<citizenHash>"]}'
```

## Network Configuration Files

| File | Purpose |
|---|---|
| `blockchain/network/configtx.yaml` | Channel policies, org definitions, orderer config |
| `blockchain/network/crypto-config.yaml` | Cryptogen spec — org structure, peer/user counts |
| `blockchain/network/crypto-config/` | Generated TLS certs and MSP material |
| `blockchain/network/channel-artifacts/` | genesis.block + channel tx — generated by `configtxgen` |

## Smoke Testing the Network

```bash
bash blockchain/scripts/invoke-test.sh
```

This sends test transactions to all 4 chaincodes and prints pass/fail for each. Safe to re-run — uses timestamp-based IDs.
